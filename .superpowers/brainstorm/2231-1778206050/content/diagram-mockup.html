Traditional REST API vs MCP Architecture

Each layer maps directly — your existing .NET skills transfer

Traditional REST API

HTTP Client

Browser, mobile app, or API consumer

↓ HTTP Request + Bearer Token

IdentityServer (OIDC)

Issues JWT access tokens (client credentials / PKCE)

↓ JWT validated by middleware

ASP.NET Core REST API

Controllers, routes, [Authorize] attribute

↓ calls

Application Services

Business logic, validation, orchestration

↓ via interfaces

EF Core

DbContext, repositories, LINQ queries

↓ SQL queries

Database

Positions, HiringOrgs, Remuneration tables
e.g. MS SQL Server, Oracle

equivalent
MCP Architecture (.NET 10)

MCP Client

Claude Desktop · VS Code Copilot · Console App · SPA · Server-side Web
SPA: Angular, Blazor WASM  |  Server-side: Razor Pages, Blazor Server

≈ HTTP Client
↓ natural language prompt

LLM (Ollama / Claude API)

Decides which MCP tools to call and with what arguments
IChatClient abstraction — swap models without changing code

≈ routing / controller selection logic
↓ JSON-RPC 2.0 tool call (MCP)

IdentityServer (OIDC)

Agent acquires token via client credentials flow or PKCE

≈ same IdentityServer
↓ Bearer token attached to MCP request

HrMcp.McpServer (ASP.NET Core)

[McpServerTool] methods · JWT middleware · HTTP/SSE or stdio
Microsoft.Extensions.AI + MCP SDK — both GA

≈ REST Controller + [Authorize]
↓ calls

Application Services

PositionService, HiringOrgService — unchanged from REST

≈ same Application Services
↓ via interfaces

EF Core

Same DbContext and repositories — zero changes

≈ same EF Core
↓ SQL queries

Database

Same database — AI never touches it directly
e.g. MS SQL Server, Oracle

≈ same Database
Key insight: Everything below the server layer — Application Services, EF Core, and your database (MS SQL, Oracle, etc.) — is identical in both architectures. You are adding an MCP surface on top of existing .NET patterns, not replacing them.
